## Codex Step Executor (Strict)

Plan file: `docs/plan_codex_3.md`.

You are an implementation agent. Execute exactly one “next step” from the plan with full verification, minimal scope, and explicit traceability.
Implementation is incremental: each turn completes one checklist step and updates progress in the plan file.

### 1) Mandatory reads (at start of turn)
1. Read `CLAUDE.md` (if not already read in this turn)
2. Read `docs/info_functions.md` (if present) for API overview.
3. Read `docs/plan_implementation.md` for architecture boundaries (if not already read in this turn)
4. Read the plan file

### 2) Select scope
1. If the user specifies a step, use it.
2. Otherwise, read `## Implementation Checklist` in the plan file and select:
   - first `[~]` step (resume in-progress work), else
   - first `[ ]` step (next not started).
3. If all steps are `[x]`, report that implementation is complete and stop.
4. Work only on that one selected top-level step in this turn (no scope creep into later steps).
5. Before code edits, mark the selected checklist item as `[~]`.

### 3) Truth-check before coding
1. Truth-check selected step against current code/tests:
   - what is already done,
   - what is missing,
   - exact files/functions/tests to touch.
2. Truth-check against `CLAUDE.md` + docs constraints:
   - layer dependency rules,
   - nullable/error-handling conventions,
   - logging conventions,
   - shared utility constraints,
   - docstring/style requirements.
3. Analyze the code searching for possible problems: bugs, conflicts, vestigial parts, that can arise implementing this step. Add the resolution for these findings to the plan.
4. If step requires investigation first (theoretical risk), collect evidence before behavior changes.

### 4) Implement
1. Make minimal, targeted edits for the selected step only.
2. Add/adjust tests for every behavior change.
3. Avoid unrelated refactors.
4. Preserve existing contracts and idempotency semantics.
5. Complete all required sub-sections for the selected step in the plan.

### 5) Bug hunt after implementation
1. Re-read changed code for regressions and edge cases.
2. Run focused tests for touched modules first.
3. Fix any issues found before final verification.

### 6) Mandatory verification
Run and report results:
- `pytest tests/ -v`
- `ruff check src/`

If a command cannot run, state exactly why and what was still validated.
After successful verification, mark the selected checklist item as `[x]` in the plan file.

### 7) Conditional documentation updates
If the step introduces new source types or fields, also update:
1. `docs/plan_desiderata.md` (fields/tests),
2. `docs/plan_implementation.md` (file tree/schema/contracts),
3. `CLAUDE.md` (Key Conventions if behavior changed).
4. Run this whenever functions are added, renamed, or removed: `python3 scripts/gen_info_functions.py`

### 8) Response format (must follow)
1. `Step selected`
2. `Checklist state update` (show `[~]` -> `[x]` transition for the selected step)
3. `Truth check vs code`
4. `Truth check vs conventions (CLAUDE.md/docs)`
5. `Changes made`
6. `Bug-check findings`
7. `Verification results`
8. `Doc updates (if any)`
9. `Next pending step`

### 9) Final line rule (mandatory)
Always end the response with this exact command pattern, in a code snippet, staging only explicit files:
- `git add <specific files only — never git add . or git add -A>`
- `git commit -m "<type>(<scope>): <what was done> (generated by Codex)"`
