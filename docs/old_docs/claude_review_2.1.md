### H1 — `no_text` behavior contradiction → **CONFIRMED, expanded**

**Codex verdict:** High.
**This review:** **Critical** (upgraded). The contradictions are more extensive than Codex identified.

**All conflicting statements, traced precisely:**

| Location | Says |
|---|---|
| `plan_desiderata.md:246` | "PDF with zero extractable text → `text_quality = no_text`, **0 `position_rows`**, `pdf_fetch_status = parse_error`." |
| `plan_desiderata.md:269` | "`ocr_degraded` or `no_text` rows have `parse_confidence = low`." (Implies rows EXIST for `no_text`.) |
| `plan_desiderata.md:99` | "Do not produce `position_rows` for failed calls." (If `no_text` is a failure, no rows.) |
| `implement_step5.md:765` | "If `text_quality` is `'no_text'` or `'ocr_degraded'`, **still attempt** segmentation and extraction … `score_confidence` will assign `low`. **Do not short-circuit.**" |
| `implement_step5.md:766` | "If `text` is empty, return `([], None)`." |
| `implement_step7.md:110` | "`build_rows()` may return `[]` when `text_quality` is `'no_text'` or `'ocr_degraded'`." (Admits they CAN return empty.) |

**The contradiction chain:**
1. Desiderata line 246 says: `no_text` → 0 rows + `parse_error`.
2. Desiderata line 269 says: `no_text` rows have `parse_confidence = low` (rows must exist).
3. Step 5 says: always attempt extraction for `no_text` (rows may exist with `low` confidence).
4. Desiderata line 99 says: don't produce rows for failures.

**Additional complication:** Step 7.2.1 (pipeline sync) does NOT set `pdf_fetch_status = "parse_error"` for `no_text` — it only sets `parse_error` when `extract_text()` raises `RuntimeError`. A `no_text` classification is a successful return from `extract_text()` (no exception), so the pipeline sets `pdf_fetch_status = "ok"`. This contradicts desiderata line 246 which says `no_text` → `parse_error`.

**Resolution needed — choose one:**
- **Option A:** `no_text` = operational success, data absence. Keep `pdf_fetch_status = "ok"`, produce 0 rows (empty text → `build_rows` returns `([], None)`), no `parse_confidence` assigned. Update desiderata line 269 to exclude `no_text` from "rows have low confidence."
- **Option B:** `no_text` = parse failure. Set `pdf_fetch_status = "parse_error"` in the pipeline when `text_quality == "no_text"`, produce 0 rows. Update step 5 to short-circuit on `no_text`. Remove the "still attempt extraction" note.
- **Option C:** `no_text` = produce 1 row with all-None fields and `parse_confidence = "low"`. Update desiderata line 246 to say "1 position_row with low confidence" instead of "0 position_rows." Keep `pdf_fetch_status = "ok"`.

Whichever option is chosen, update ALL affected locations consistently.

### M1 — Detail-fetch error → `pdf_fetch_status` semantic confusion → **CONFIRMED, expanded**

**Codex verdict:** Medium.
**This review:** **High** (upgraded). The confusion runs deeper than Codex identified.

`implement_step4.md:281`: "On HTTP error fetching a detail page: log at WARNING, set `call.pdf_fetch_status = 'download_error'`, and continue."

**Three problems:**

1. **Semantic misuse:** `pdf_fetch_status` should reflect PDF-related outcomes, not HTML detail page fetch outcomes. The values `ok | download_error | parse_error | skipped` are defined in context of PDF processing (`plan_desiderata.md:99,175`).

2. **Architectural impossibility:** If the detail page HTTP request fails, `parse_detail()` is never called, so no `CallRaw` object is created. The orchestrator can't "set `call.pdf_fetch_status`" on an object that doesn't exist yet. The orchestrator would need to create a minimal `CallRaw(detail_id=..., pdf_fetch_status="download_error")` using only the `detail_id` from the listing row — but this isn't specified.

3. **Data quality:** A call with `pdf_fetch_status = "download_error"` but where the HTML page (not the PDF) failed gives misleading analytics. The user would think the PDF download failed when actually the detail page was unreachable.

**Resolution needed:** Either:
- Add a separate field (e.g., `detail_fetch_error: bool`) to `CallRaw` for detail page failures, OR
- Skip the call entirely on detail page failure (log + continue), never creating a `CallRaw` for it. The next sync will retry.

## New Findings

### N1. High — `text_quality` type mismatch: enum vs string

`extract_text()` returns `tuple[str, TextQuality]` where `TextQuality` is an enum (`implement_step5.md:77`).

`build_rows()` accepts `text_quality: str` (`implement_step5.md:749`).

`score_confidence()` accepts `text_quality: str` and compares against `TextQuality.*.value` (`implement_step5.md:659`).

The pipeline (`implement_step7.md:96`) does:
```python
text, text_quality = extract_text(pdf_path)
rows, pdf_call_title = build_rows(text, call.detail_id, text_quality, anno)
```

Here `text_quality` is a `TextQuality` enum, but `build_rows` expects a `str`. Python won't crash (since `TextQuality` extends `str` via `class TextQuality(str, Enum)`), so this works at runtime because `TextQuality.DIGITAL == "digital"` is `True`. However:

- The type annotations are misleading — `str` is not `TextQuality`.
- `PositionRow.text_quality` is `str | None`, and the stored value must be the `.value` string, not the enum member.
- `score_confidence` compares `text_quality` against `TextQuality.OCR_DEGRADED.value` — if it receives an enum, `text_quality == TextQuality.OCR_DEGRADED.value` works because `str(Enum)` coercion, but `text_quality == "ocr_degraded"` also works because `TextQuality(str, Enum)`.

**Impact:** No runtime crash due to `str` inheritance in the enum, but the type signatures are inconsistent and could cause subtle bugs if someone changes the enum base class.

**Resolution:** Decide on one approach:
- Pass `text_quality.value` (str) at the pipeline boundary, keeping downstream as `str`, OR
- Change `build_rows` and `score_confidence` signatures to accept `TextQuality` enum.

### N2. High — No rate limiting for PDF downloads

`CLAUDE.md`: "Sleep 1.0 s between every request (active + expired listings, detail pages, PDF downloads)."

`implement_step4.md:278`: The orchestrator sleeps between listing page and detail page fetches.

`implement_step5.md:44-61` (`downloader.py`): No mention of rate limiting or sleeping.

`implement_step7.md:114`: "Rate limiting: enforced inside `fetch_all_calls` and `download`."

But `download()` as specified in step 5.1.1 has **no sleep logic**. The function signature is `download(url, dest, force)` — it downloads immediately. There is no sleep between consecutive PDF downloads in the pipeline loop.

**Impact:** The scraper will fire PDF download requests as fast as the network allows, violating the 1.0s rate limit policy. This could trigger server-side rate limiting or bans.

**Resolution:** Either:
- Add `time.sleep(RATE_LIMIT_SLEEP)` before the HTTP GET inside `download()`, OR
- Add the sleep in the pipeline loop (`run_sync`) before each `download()` call, OR
- Pass the configured `Session` from `get_session()` to `download()` and add sleep there.

### N3. Medium — `download()` does not use the configured session

`implement_step5.md:56`: "Use a `requests.Session` (or plain `requests.get`) to download. Pass the session from the caller if available, or create a short-lived one."

But the function signature is `download(url: str | None, dest: Path, force: bool = False)` — no `session` parameter.

The pipeline (`implement_step7.md:62-63`) imports both `get_session` and `download` but only passes the session to `fetch_all_calls`, not to `download`.

**Impact:** PDF downloads won't benefit from the retry configuration (3 retries with exponential backoff on 5xx) set up in `get_session()`. A transient 503 during PDF download will cause an immediate failure instead of retrying.

**Resolution:** Either:
- Add `session: requests.Session | None = None` parameter to `download()`, OR
- Have `download()` create its own session using the same retry config from `get_session()`.

### N4. Medium — `_PROJECT_ROOT` calculation error in Write section

`implement_step3.md:31` (Write section):
```python
_PROJECT_ROOT: Path = Path(__file__).parent.parent.parent
```

`implement_step3.md:43` (Notes section, same sub-substep):
> "Correct chain: `Path(__file__).parent` = `config/`, `.parent` = `infn_jobs/`, `.parent` = `src/`, `.parent` = project root. So use `Path(__file__).parent.parent.parent.parent`"

The Write section has **three** `.parent` calls but the Notes section correctly identifies **four** are needed. An implementer following the Write section will get `_PROJECT_ROOT = src/` instead of the actual project root, causing `DB_PATH`, `EXPORT_DIR`, and `PDF_CACHE_DIR` to resolve to `src/data/...` instead of `data/...`.

**Resolution:** Fix the Write section in `implement_step3.md:31` to use `.parent.parent.parent.parent`.

### N5. Medium — Test coverage gaps for critical modules

The following modules have **no dedicated test file** — they are covered only by the e2e test in Step 9, which mocks the I/O boundaries:

| Module | Step | Risk |
|---|---|---|
| `extract/pdf/downloader.py` | 5.1 | Cache logic, error handling untested |
| `extract/parse/fields/metadata.py` | 5.10 | `extract_pdf_call_title` logic untested |
| `extract/parse/fields/confidence.py` | 5.14 | Confidence rubric untested in isolation |
| `extract/parse/row_builder.py` | 5.16 | Integration of all extractors untested |
| `fetch/orchestrator.py` | 4.7 | Listing → detail → assembly flow untested |

The e2e test (`implement_step9.md`) mocks `fetch_all_calls`, `download`, and `extract_text` — so the orchestrator, downloader, and mutool wrapper are **never exercised** even in e2e. The row_builder IS exercised via the unmocked `build_rows` call, but only with one fixture (`multi_same_type.txt`).

**Impact:** Bugs in the download cache logic, the confidence rubric, or the orchestrator's assembly flow won't be caught until a live run. Given that the confidence rubric has three tiers with specific conditions (H1 contradictions aside), this is risky.

**Resolution:** Add test files for at least `confidence.py` and `row_builder.py`. These are pure-logic modules that don't need I/O mocking — just inline data. A `test_confidence.py` with 5-6 tests covering each tier, and a `test_row_builder.py` using the existing fixtures, would significantly improve coverage.

### N6. Low — `parse_date()` appears unused in v1

`parse_date()` is defined in `implement_step5.md:158` and tested in `implement_step5.md:233-252`, but no field extractor in step 5 calls it. Dates in `CallRaw` (`data_bando`, `data_scadenza`) come from the HTML detail parser as raw strings. No PDF field extractor extracts dates.

**Impact:** The function and its tests are implemented but never called, adding dead code. This isn't harmful but wastes implementation effort.

**Resolution:** Either:
- Document it as a utility for future v2 use and keep it, OR
- Remove it from step 5 scope and add it when needed, OR
- Use it in the detail parser (step 4) to validate/normalize HTML date strings before storing — but this would change the detail parser's scope.

### N7. Low — SQLite foreign key enforcement not enabled

`implement_step6.md:37`: `position_rows` declares `FOREIGN KEY (detail_id) REFERENCES calls_raw(detail_id)`.

`implement_step6.md:199`: The delete/insert order for curated tables is carefully specified because of FK constraints.

But SQLite does **not** enforce foreign keys by default. `PRAGMA foreign_keys = ON` must be executed on each connection. Neither `schema.py` (`implement_step6.md:26-42`) nor `conftest.py` (`implement_step6.md:53-67`) nor `cmd_sync.py` (`implement_step8.md:114-121`) mention this pragma.

**Impact:** The FK declarations are decorative — they won't prevent orphaned `position_rows` or curated table inconsistencies. The careful delete/insert ordering in `rebuild_curated()` is correct but unnecessary without the pragma. If FK enforcement is later enabled, existing data may violate constraints.

**Resolution:** Either:
- Add `conn.execute("PRAGMA foreign_keys = ON")` after `sqlite3.connect()` in `cmd_sync.py` and `cmd_export.py`, OR
- Document that FKs are intentionally unenforced (schema-as-documentation) and remove the ordering requirements from `rebuild_curated()`.
